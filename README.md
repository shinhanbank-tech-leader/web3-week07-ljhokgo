# 스마트 컨트랙트 업그레이드 방식과 은행 시스템 변경 방식 비교 분석

---

## 1. 핵심 개요

소프트웨어는 반드시 변한다. 버그가 발견되고, 비즈니스 요건이 바뀌고, 규제가 개정된다. 은행 IT 시스템은 수십 년간 체계적인 변경 관리(Change Management) 프로세스를 통해 이 변화를 수용해왔다. 반면 스마트 컨트랙트는 **"배포된 코드는 불변이다"**라는 블록체인의 핵심 속성 위에서 작동한다. 이 근본적 차이가 업그레이드 방식, 리스크 구조, 운영 통제 체계 전반에 걸쳐 전혀 다른 패러다임을 만들어낸다.

---

## 2. 왜 업그레이드가 어려운가: 스마트 컨트랙트의 불변성

이더리움에 배포된 스마트 컨트랙트는 그 주소에 영구적으로 고정된다. 코드 수정이 필요하면 **새로운 컨트랙트를 새 주소에 재배포**해야 한다. 기존 컨트랙트는 그대로 남아 여전히 호출 가능한 상태로 존재한다.

이것이 문제가 되는 이유는 세 가지다.

- **자산과 상태의 이전**: 기존 컨트랙트에 묶인 사용자 자산, 잔액, 포지션 데이터를 새 컨트랙트로 옮겨야 한다
- **외부 참조 업데이트**: 기존 컨트랙트 주소를 직접 하드코딩한 다른 컨트랙트·프론트엔드·시스템이 모두 새 주소를 참조하도록 변경해야 한다
- **사용자 신뢰**: "코드가 바뀌지 않는다"는 신뢰를 기반으로 자산을 예치한 사용자에게 업그레이드는 신뢰 위반이 될 수 있다

---

## 3. 스마트 컨트랙트 업그레이드 패턴

불변성을 우회하면서 사실상의 업그레이드를 가능하게 하는 설계 패턴들이 존재한다.

### 3-1. 프록시 패턴 (Proxy Pattern)
가장 널리 사용되는 방식이다. 사용자는 항상 **프록시 컨트랙트(주소 고정)**와 상호작용하고, 실제 로직은 **구현 컨트랙트(Logic Contract)**에 분리한다. 업그레이드 시 프록시가 가리키는 구현 컨트랙트 주소만 교체한다.

- **장점**: 사용자 주소·자산 유지, 로직만 교체 가능
- **리스크**: 프록시 관리자 키 탈취 시 로직 전체를 악의적 코드로 교체 가능. 실질적으로 중앙화된 통제권 발생
- **대표 사례**: OpenZeppelin TransparentUpgradeableProxy, UUPS

### 3-2. 다이아몬드 패턴 (EIP-2535)
하나의 프록시가 여러 개의 기능별 구현 컨트랙트(Facet)를 참조한다. 전체 교체 없이 특정 기능(Facet)만 선택적으로 업그레이드 가능하다. 은행 시스템의 **모듈별 패치 배포**와 가장 유사한 구조다.

### 3-3. 데이터-로직 분리 패턴
Storage 컨트랙트(상태·데이터)와 Logic 컨트랙트(비즈니스 로직)를 처음부터 분리 설계한다. 로직 교체 시 데이터 이전 없이 Storage 컨트랙트의 로직 참조만 변경한다.

### 3-4. 마이그레이션 (재배포)
업그레이드 패턴 없이 새 컨트랙트를 배포하고 사용자가 직접 자산을 이전하도록 유도하는 방식. 가장 단순하지만 사용자 경험이 나쁘고, 이전 기간 동안 구버전과 신버전이 공존하는 혼란이 발생한다.

## 4. 은행 시스템의 변경 관리 방식

은행 IT의 변경 관리는 금융감독원 IT 검사 기준, ISO 20000, ITIL 프레임워크를 기반으로 **고도로 제도화된 프로세스**를 따른다.

### 4-1. 변경 관리 프로세스 단계

| 단계 | 내용 | 담당 |
|---|---|---|
| 변경 요청 (RFC) | 변경 목적·영향도·롤백 계획 포함 문서 작성 | 개발팀 |
| 영향도 분석 | 연관 시스템·인터페이스·데이터 영향 검토 | 아키텍처팀 |
| 변경 승인 | CAB(변경자문위원회) 또는 IT 위원회 심의·승인 | IT 거버넌스 |
| 개발·테스트 | 개발 → 단위테스트 → 통합테스트 → UAT | 개발·QA팀 |
| 스테이징 배포 | 운영 환경과 동일한 스테이징에서 검증 | 운영팀 |
| 운영 배포 | 배포 후 모니터링, 이상 시 즉시 롤백 | 운영팀 |
| 사후 검토 | 변경 결과 기록, 예상치 못한 영향 분석 | 전체 |

### 4-2. 롤백(Rollback) 체계
은행 시스템 배포의 핵심 안전장치는 **즉각적 롤백 가능성**이다. 배포 후 이상 징후 발견 시 이전 버전으로 수분~수십 분 내 복구가 가능하도록 롤백 절차가 표준화되어 있다. DB 스키마 변경의 경우 역방향 마이그레이션 스크립트를 사전 준비한다.

### 4-3. 블루-그린 배포·카나리 배포
대형 시스템 변경 시 구버전(Blue)과 신버전(Green)을 동시에 운영하다가 트래픽을 점진적으로 전환하는 무중단 배포 방식을 사용한다. 문제 발생 시 트래픽을 즉시 구버전으로 되돌릴 수 있다.

## 5. 핵심 비교 분석

| 비교 항목 | 스마트 컨트랙트 | 은행 시스템 |
|---|---|---|
| 기본 변경 방식 | 새 컨트랙트 재배포 또는 프록시 교체 | 현행 시스템 직접 패치·버전 업 |
| 즉각 롤백 | 불가 (배포된 코드는 영구 존재) | 가능 (표준 롤백 절차 존재) |
| 변경 승인 주체 | 컨트랙트 소유자 키 또는 거버넌스 투표 | CAB·IT위원회·경영진 승인 |
| 변경 이력 관리 | 온체인 트랜잭션으로 자동·불변 기록 | Git·ITSM 티켓·배포 로그 |
| 긴급 패치 속도 | 느림 (재배포·감사·검증 필요) | 빠름 (핫픽스 즉시 배포 가능) |
| 버그 발견 시 피해 | 배포 후 즉시 악용 가능, 되돌리기 불가 | 롤백으로 피해 최소화 가능 |
| 변경 투명성 | 완전 공개 (누구나 코드·이력 열람) | 내부 접근 통제 (감사인만 열람) |
| 사용자 영향 | 자산 이전·주소 변경 등 직접 개입 필요 | 서비스 중단 최소화, 사용자 무관 |
| 규제 감사 | 온체인 이벤트로 자동 감사 가능 | ITSM·배포 문서 기반 감사 |
| 업그레이드 권한 | 키 보유자에 집중 (중앙화 리스크) | 역할 기반 접근 통제(RBAC) |

## 6. 업그레이드 방식이 만드는 운영·통제 리스크

### 스마트 컨트랙트의 고유 리스크

**업그레이드 키 탈취**
프록시 패턴에서 구현 컨트랙트 교체 권한을 가진 관리자 키가 탈취되면, 공격자는 전체 로직을 악의적 코드로 교체할 수 있다. 2022년 Ronin Bridge 해킹(약 6,200억 원)은 다중서명 키 탈취로 발생한 대표 사례다. 이 리스크는 탈중앙화를 표방하는 프로토콜이 실질적으로 중앙화된 통제권을 내포하고 있음을 보여준다.

**감사(Audit)가 배포 전에 완결되어야 함**
은행 시스템은 운영 중 긴급 핫픽스가 가능하지만, 스마트 컨트랙트는 배포 후 수정이 불가능하므로 **코드 감사가 배포 전에 완전히 완료**되어야 한다. 이는 개발-배포 사이클을 길게 만들고, 긴급 보안 패치 대응을 구조적으로 어렵게 한다.

**거버넌스 지연**
DAO 방식의 온체인 거버넌스(토큰 투표)로 업그레이드를 결정하는 프로토콜은 긴급 상황에서도 투표 기간(수 일~수 주)을 거쳐야 한다. 이 기간 동안 알려진 취약점이 악용될 수 있다.

### 은행 시스템의 고유 리스크

**변경 실패로 인한 대규모 장애**
2011년 RBS(영국 로열뱅크오브스코틀랜드) IT 장애, 국내에서도 간헐적으로 발생하는 코어뱅킹 점검 중 장애는 변경 관리 실패가 전체 서비스 중단으로 이어질 수 있음을 보여준다. 다만 롤백 체계로 피해를 제한할 수 있다.

**섀도 IT·기술 부채 누적**
수십 년간 패치를 반복한 레거시 코어뱅킹은 변경의 영향 범위를 예측하기 어렵다. 단순한 화면 수정이 예상치 못한 다운스트림 시스템 장애로 이어지는 사례가 반복된다.

## 7. 은행 IT 관점의 스마트 컨트랙트 변경 관리 적용 방안

스마트 컨트랙트를 금융 서비스에 도입할 때, 기존 은행 변경 관리 프로세스를 다음과 같이 적용해야 한다.

**배포 전**
- 외부 보안 감사(Audit) 의무화: 배포 전 최소 2개 이상의 독립 기관 감사 완료
- 형식 검증(Formal Verification): 수학적 방법으로 코드 동작의 정확성 증명
- 스테이징 네트워크(테스트넷) 충분한 검증 기간 확보

**배포 설계**
- 다중서명(Multisig) 업그레이드 키: 단일 담당자가 아닌 3-of-5 이상의 다중서명으로 업그레이드 권한 분산
- 타임락(Timelock) 적용: 업그레이드 실행 전 48~72시간의 지연 시간을 강제, 이해관계자가 이상을 발견하고 대응할 기회 확보
- 긴급 정지(Circuit Breaker) 설계: 이상 징후 감지 시 컨트랙트 기능을 일시 정지하는 비상 스위치 내장

**배포 후**
- 온체인 이벤트 실시간 모니터링: 비정상적 자금 흐름, 대규모 인출 감지
- 인시던트 대응 계획: 스마트 컨트랙트 특성상 롤백이 불가능하므로, 피해 최소화를 위한 사전 정의된 대응 절차(예치금 이전, 서비스 일시 중단) 수립

## 8. 결론

> **은행 시스템의 변경 관리는 "변경을 안전하게 적용하는 방법"을 고민하고, 스마트 컨트랙트의 변경 관리는 "변경할 수 없는 코드를 처음부터 올바르게 만드는 방법"을 고민한다.**

두 방식의 본질적 차이는 **실수의 비용**에 있다. 은행 시스템은 롤백이라는 안전망이 있어 실수를 되돌릴 수 있지만, 스마트 컨트랙트는 배포된 순간부터 실수가 고정되어 악용될 수 있다. 이것이 스마트 컨트랙트를 금융 서비스에 도입할 때 개발 품질 관리와 배포 설계에 은행 시스템보다 훨씬 높은 수준의 엄격함이 요구되는 이유다.

금융 IT 관점의 실용적 결론은 다음과 같다. 변경이 잦고 비즈니스 요건이 유동적인 영역(고객 인터페이스, 상품 조건 변경)은 기존 은행 시스템이 유리하고, 변경 빈도가 낮고 실행의 투명성·자동화가 중요한 영역(정산, 이자 지급, 담보 청산)은 스마트 컨트랙트가 유리하다. 두 방식의 강점을 용도에 맞게 조합하는 **하이브리드 변경 관리 체계**가 국내 금융권의 현실적인 도입 전략이다.
